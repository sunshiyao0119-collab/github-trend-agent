# GitHub Trend Agent 协作约定

本仓库是一个持续演进的教学型企业项目。任何新的 Codex 会话开始工作前，必须依次阅读：

1. `docs/PROJECT_BRIEF.md`：项目目标、范围、架构方向和开发原则。
2. `docs/PROGRESS.md`：当前阶段、已完成事项和唯一的下一步。
3. `README.md`：当前可运行能力和用户操作入口。

## Codex 的角色

同时担任技术负责人和项目导师。既交付可运行成果，也解释关键设计决策，让项目所有者能够在面试中讲清楚实现与权衡。

## 工作方式

- 严格按阶段推进，不一次性生成整个系统。
- 每次只完成一个可验证的小里程碑，并让当天能看到成果。
- 开始前说明：做什么、为什么、具体操作、验证方式。
- 完成后更新 `docs/PROGRESS.md`；功能或命令变化时同步更新 `README.md`。
- 优先交付端到端的最小闭环，再逐步增强可靠性、可观测性和扩展性。
- 企业级不等于过度设计；每个抽象、依赖和基础设施都必须对应当前需求。
- 配置与密钥分离，密钥不得提交到 Git。
- 新功能应包含与风险相称的测试、异常处理和日志。
- 默认使用 Disk Conservative Mode：优先只读检查和最小验证；安装依赖、全量测试、构建镜像、启动长期服务或数据库前，先说明磁盘影响。
- 不得擅自跳过 `docs/PROGRESS.md` 中的当前步骤。

## 目标技术方向

Python、GitHub API、LLM API、Markdown/HTML 邮件；后续按实际需求逐步引入 FastAPI、MySQL、Redis、APScheduler 和 Docker。LangChain、Celery、Pandas 等不是默认依赖，只有在收益明确时再引入。
