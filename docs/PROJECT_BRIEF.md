# GitHub Trend Agent 项目宪章

## 1. 项目定位

GitHub Trend Agent（GitHub 技术趋势智能分析助手）是一个可持续成长的企业级 AI 应用项目。它每天发现值得关注的 GitHub 项目，完成清洗、评分和大模型分析，生成技术日报并发送邮件；后续扩展历史趋势分析、问答 Agent 和 Web Dashboard。

项目的双重目标是：

1. 交付一个真实可用、可部署、可写进简历的系统。
2. 系统学习 Python 工程、后端、ETL、LLM 应用和部署能力，并能解释关键技术决策。

## 2. 核心用户价值

将人工浏览 GitHub Trending、GitHub Search 等渠道的重复工作，转化为稳定的自动化日报：

- 发现热门、快速增长、AI 相关和优秀 Python 项目。
- 解释项目做什么、为什么值得关注、技术与学习价值是什么。
- 保存历史数据，以便观察趋势而不只是展示静态 Star 总数。

## 3. 演进路线

### V1：最小端到端闭环

GitHub API 采集 → 数据清洗 → 可解释评分 → LLM 分析 → Markdown/HTML 日报 → SMTP 邮件。

### V2：历史数据

存储项目快照和报告，支持真实的 Star 增长速度与趋势计算。数据库方案在进入该阶段时根据部署环境决定；MySQL 是目标生产方案，本地开发可评估轻量替代方案。

### V3：自动运行

使用 APScheduler 或平台定时任务每日执行，补充幂等、重试、失败告警和运行记录。

### V4：工具化 Agent

提供 GitHub 搜索、项目分析、趋势分析和报告生成工具，支持自然语言问题，例如“最近 AI Agent 领域有哪些热门项目？”

### V5：Web 产品

FastAPI 提供 API，前端 Dashboard 展示今日热门项目、技术分类、趋势变化和历史报告。

### V6：部署与工程强化

Docker 化、CI、生产配置、监控、安全检查和性能优化。

## 4. V1 数据契约（初稿）

```json
{
  "name": "owner/repository",
  "description": "project description",
  "language": "Python",
  "stars": 0,
  "forks": 0,
  "url": "https://github.com/owner/repository",
  "topics": [],
  "owner": "owner",
  "updated_at": "ISO-8601 timestamp",
  "pushed_at": "ISO-8601 timestamp"
}
```

契约会在采集阶段用类型模型正式定义。字段调整必须说明兼容性影响。

## 5. 评分原则

评分必须可解释、可测试，并区分“受欢迎程度”和“增长趋势”。在没有历史快照前，不能声称准确计算 Star 增长速度；V1 先使用 GitHub API 当下可获得的代理指标，V2 再用历史数据替换。

初始候选因素包括：

- Star 与 Fork 的对数归一化结果，避免头部项目完全支配排名。
- 最近更新时间或活跃度。
- 与关注主题、语言的匹配度。
- 引入历史快照后计算的时间窗口增长率。

具体权重在评分阶段通过小样本验证后确定，不提前写死。

## 6. 阶段计划

0. 项目初始化：Git 仓库、Python 环境、目录骨架、配置管理。
1. GitHub 数据采集：鉴权、请求、限流、数据模型和最小测试。
2. 清洗与评分：去重、缺失值处理、过滤、可解释排名。
3. LLM 分析：供应商抽象、Prompt、结构化输出、成本与失败降级。
4. 日报与邮件：Markdown、HTML、SMTP、安全配置。
5. 数据库与历史分析：表设计、迁移、快照和趋势。
6. FastAPI：查询 API、校验、错误响应和接口文档。
7. 自动化与 Docker：调度、部署、CI 和运行手册。

## 7. 开发完成标准

一个小里程碑只有同时满足以下条件才算完成：

- 有明确的用户可见结果或工程能力。
- 最小验证通过，并记录验证命令或证据。
- 不包含密钥、个人邮箱等敏感信息。
- README 和进度记录与实际状态一致。
- 项目所有者能够说明“为什么这样设计”。
